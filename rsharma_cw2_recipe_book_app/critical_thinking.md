1) Flutter's Navigator works like a stack of cards. When you call Navigator.push(), a new screen gets placed on top, and when the user taps back, Navigator.pop() removes it. In my app, when it starts the stack only has HomeScreen. When the user taps on a recipe card I call Navigator.push() with a MaterialPageRoute to open DetailsScreen. At this point the stack has two screens: HomeScreen at the bottom and DetailsScreen on top. The user sees the details but HomeScreen is still alive underneath. When the user presses the back arrow Flutter automatically calls Navigator.pop(), which removes DetailsScreen from the stack and brings the user back to HomeScreen. Since I pass the whole Recipe object through the constructor when pushing DetailsScreen, I don't need to worry about looking up data on the second screen.

2) A StatelessWidget is a widget that does not change after it's built. It takes in data and displays it, but it can't update itself. A StatefulWidget on the other hand has a State object that can change over time, and when you call setState() Flutter rebuilds the widget to reflect those changes. In my app I used StatelessWidget for both HomeScreen and DetailsScreen because neither screen needs to change its own state. HomeScreen just reads from the sampleRecipes list and displays the data. DetailsScreen receives a Recipe object through its constructor and displays the name, image, ingredients, and instructions — none of which change while the user is on the screen. My rule of thumb is if a screen only displays data that doesn't change while the user is looking at it I use StatelessWidget. If the screen needs to update something based on user interaction like toggling a favorite or handling form input then I would use StatefulWidget.

3) In my app I created a Recipe class in recipe.dart that has fields for name, imagePath, ingredients, and instructions. I then created a separate file called recipes_data.dart that holds a list of Recipe objects with all the actual data. This is a model-driven approach. The alternative would have been to hard-code the recipe text directly inside DetailsScreen using if statements or switch cases to show different content based on some index. But that approach would be messy and hard to maintain. If I were building a production app that fetched recipes from a REST API I would mainly need to change the data layer. Instead of a static list I would use the http package to make a GET request and parse the JSON response into Recipe objects. The model class would stay mostly the same — I would just add a fromJson() factory constructor. The screens wouldn't need to change much since they already work with Recipe objects.

4) Adding an image asset to a Flutter app involves several steps. First I created an assets/images/ folder and placed my recipe photos there. Second I declared the assets in pubspec.yaml under the flutter section. Third I ran flutter pub get so Flutter knows about the new assets. Finally in my code I used Image.asset() to display them. Assets must be declared in pubspec.yaml because Flutter needs to know which files to bundle into the app at build time. Two pitfalls I can think of are incorrect indentation in pubspec.yaml since YAML is very sensitive to spacing, and file paths being case-sensitive — if the actual file is Pasta.PNG but the code says pasta.png it might work on some systems during development but fail on others.

5) For my HomeScreen I decided to use a ListView.builder with Card widgets containing ListTile. I chose this layout because it gives a clean organized list where each recipe has a small image on the left, the name in the middle, and a chevron arrow on the right that signals the item is tappable. For DetailsScreen I placed a large hero image at the top using Image.asset with width: double.infinity and BoxFit.cover so it stretches across the full width. Below that I organized the content into clear sections — the recipe name in bold, then ingredients with bullet points, then instructions. I wrapped everything in a SingleChildScrollView so it doesn't overflow on smaller screens. The design principles that guided my choices were visual hierarchy, consistency, and affordance.

6) In my current app the data comes from a static list in recipes_data.dart so there is no state that needs to persist between sessions. But if I had a favourites feature the favourites list would reset every time the app restarts because it only lives in memory. To make data persist one approach is using shared_preferences which stores simple key-value pairs on the device. It's lightweight and easy to set up. The second approach is using a local database like sqflite which gives a full SQLite database on the device where I can create tables, run queries, and store structured data. For a simple favourites feature shared_preferences would be enough but if the app grew to include user-created recipes, ratings, or notes sqflite would be the better choice.
